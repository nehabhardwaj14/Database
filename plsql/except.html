EXCEPTION HANDLING

Exception handling in PL/SQL is vital for gracefully managing errors that may occur during program execution. PL/SQL provides a robust mechanism for handling exceptions, ensuring that your programs can respond appropriately to unexpected situations

HANDLING EXCEPTIONS:

SYNTAX:

            BEGIN
            -- Main block of code
        EXCEPTION
            WHEN exception_name1 THEN
                -- Handling code for exception_name1
            WHEN exception_name2 THEN
                -- Handling code for exception_name2
            ...
            [WHEN OTHERS THEN
                -- Default handling code for all other exceptions]
        END;

EXAMPLE:

            DECLARE
            x NUMBER := 0;
        BEGIN
            -- Attempting division by zero
            DBMS_OUTPUT.PUT_LINE(10 / x);
        EXCEPTION
            WHEN ZERO_DIVIDE THEN
                DBMS_OUTPUT.PUT_LINE('Cannot divide by zero');
        END;

PREDEFINED EXCEPTIONS

PL/SQL provides a set of predefined exceptions that cover common error scenarios. These predefined exceptions simplify error handling by giving meaningful names to specific error conditions. Here are some of the commonly used predefined exceptions in PL/SQL:

  • NO_DATA_FOUND: Raised when a SELECT INTO statement returns no rows.

  • TOO_MANY_ROWS: Raised when a SELECT INTO statement returns multiple

  • ZERO_DIVIDE: Raised when an attempt is made to divide by zero.

  • INVALID_NUMBER: Raised when an attempt is made to convert a character string into a number, but the string does not represent a valid number.

  • LOGIN_DENIED: Raised when a login attempt fails due to invalid credentials.

  • PROGRAM_ERROR: Raised when a PL/SQL program encounters an internal error.

  • STORAGE_ERROR: Raised when a PL/SQL program runs out of memory or exceeds memory limits.

  • VALUE_ERROR: Raised when an arithmetic, conversion, or assignment operation encounters an invalid value.

  • CURSOR_ALREADY_OPEN: Raised when attempting to open a cursor that is already open.

  • INVALID_CURSOR: Raised when a cursor operation is attempted on an invalid cursor.

  • TIMEOUT_ON_RESOURCE: Raised when a timeout occurs while waiting for a resource.

  • NO_DATA_NEEDED: Raised when an implicit cursor does not need to fetch any data.

USER-DEFINED EXCEPTIONS

User-defined exceptions in PL/SQL allow you to create custom error conditions tailored to your application's specific requirements. You can also define your own custom exceptions using the DECLARE section and raise them explicitly when necessary.

SYNTAX:

                DECLARE
                exception_name EXCEPTION;
            BEGIN
                IF condition THEN
                    RAISE exception_name;
                END IF;
            END;

EXAMPLE:

                DECLARE
                my_exception EXCEPTION;
            BEGIN
                IF condition THEN
                    RAISE my_exception;
                END IF;
            EXCEPTION
                WHEN my_exception THEN
                    DBMS_OUTPUT.PUT_LINE('Custom exception occurred');
            END;

Exception Handling Best Practices:

  • Catch specific exceptions whenever possible to handle them appropriately.

  • Always include a catch-all WHEN OTHERS block for handling unexpected exceptions and logging errors.

  • Use user-defined exceptions to create meaningful error messages and handle specific error scenarios.

  • Keep exception handling code concise and focused on error recovery or graceful degradation.
